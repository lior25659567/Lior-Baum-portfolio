# UX case-study lexicon — correct names, used sparingly

> **Read `_writing-voice.md` first — it outranks this file.** The voice doctrine is:
> **concrete product detail first, UX language second, and earned, not decorative.** This
> lexicon is NOT a list of words to sprinkle across slides. It is a reference for the
> *correct name* on the **rare** occasion you actually name a method — so that when you do,
> you use the established term instead of an invented one. If a slide can be written in
> plain product language, write it in plain product language and skip the term entirely.

Overusing these terms (friction points, pain points, cognitive load, recognition over
recall, journey mapping, phase handoff, workflow optimization, opportunity areas) is the
exact failure mode the voice doctrine exists to fix — it makes the designer sound like
they're proving they know UX words instead of telling the product story. Use a term only
**after** a concrete example has earned it, never to decorate.

---

## How to use it

**Writers (author / editor / copy-writer):**
- Default to **plain product language** ("Clinicians had to stop mid-scan to find a
  tool"). Only when naming the *method itself* adds clarity, use the exact term from below
  rather than a made-up one (so "we judged the old screens ourselves against usability
  rules" can be labelled a **Heuristic Evaluation** — once, where it helps).
- **Earn it first.** Give the concrete example, then — if it still helps — name it. Never
  open with the term. Never repeat the same term across slides. Never capitalize a UX term
  mid-sentence (only slide titles get title case).
- The correct name matters most in a **slide title or a method label**, not scattered
  through body copy.

**Reviewers (ux-reviewer / recruiter / director / critic / portfolio-consistency):**
- Flag **two** failure modes, and the first is now the priority: (1) **over-used or
  unearned UX jargon** and dramatic taglines — copy that names concepts instead of telling
  the product story; recommend the plain-language rewrite. (2) On the rare occasion a method
  *is* named, an **invented / non-standard label** where an established term exists —
  recommend the correct name.
- Do NOT push to add UX terms to plain copy that already reads well. Precise-but-plain beats
  precise-and-jargony.

## Term → beat / slide mapping

| Beat / slide area | Pull terms from |
|---|---|
| Research slides (research, info, quotes) | **Research & Discovery**, **Insights** |
| Problem / context slides | **Problem Definition** |
| Journey / flow / process / timeline slides | **User Journey**, **Design Process** |
| Stats / outcomes / goals slides | **Product & Business**, **Metrics & Outcomes** |
| Analysis / reasoning anywhere | **UX Analysis** |
| Ideation / directions slides | **Ideation** |
| Testing / validation slides | **Validation & Testing** |
| Reflection / positioning / senior framing | **Advanced UX vocabulary** |

## Vague claim → concrete product detail (NOT → jargon)

The fix for a vague line is almost always a **concrete product detail**, not a UX term. Say
what actually happened; the term, if used at all, comes *after* and only when it clarifies.

| Vague / amateur | Fix with concrete detail (term optional, earned, at most once) |
|---|---|
| "users got confused / it was annoying" | say what they did: "Clinicians had to leave the scan to find a tool, which broke their rhythm." (only then, if useful: "a friction point.") |
| "it got easier / cleaner" | say what changed: "Tools now sit in one fixed place, so clinicians stop hunting for them." |
| "it worked better" | say what improved, with a real or honestly-qualified figure: "In testing, clinicians reached the scan screen without the old modal detour." Never a faked metric. |
| "the result was good" | say the observed change: "Clinicians said the clearer review state made it easier to know when the scan was ready." |
| "the problem was…" (loose) | name the actual breakdown: "The biggest problems showed up when clinicians moved from setup to scanning to reviewing — one task in the clinic, three separate areas in the system." |
| "we tested it" | say how: "In the sessions I observed…" / "During clinician feedback sessions…" — never invent counts. |

When you genuinely need to *name a method* (in a title or once in body), use the established
term from the lists above instead of an invented one — but only after the concrete detail
has earned it. Plain-and-clear beats precise-and-jargony.
